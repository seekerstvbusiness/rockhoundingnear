# /import-batch

Validates and imports a batch of enriched locations to Supabase. All imported with `published: false`.

**Usage:** `/import-batch [state-slug or file-path]`  
**Examples:**
- `/import-batch arizona` — import all validated locations for Arizona
- `/import-batch scripts/data/validated/my-batch.json` — import a specific file

## Arguments

`$ARGUMENTS` — either a state slug or a path to a JSON file containing an array of location objects

## Steps

1. **Resolve input:**
   - If `$ARGUMENTS` looks like a state slug (no `/` or `.`): read all files from `scripts/data/validated/` where the location's `state_slug` matches.
   - If `$ARGUMENTS` is a file path: read that file (expects a JSON array).

2. **Pre-import checks:**
   - Count locations to import. Print: `Preparing to import {N} locations for {state}`
   - For each location, verify slug is not already in `scripts/data/imported.log`
   - Run quick validation: confirm all 17 minimum fields present. Skip any that fail.

3. **Read Supabase credentials from `.env.local`:**
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `SUPABASE_SERVICE_ROLE_KEY`

4. **Insert in batches of 50** using the Supabase REST API:
   ```
   POST {SUPABASE_URL}/rest/v1/locations
   Authorization: Bearer {SERVICE_ROLE_KEY}
   Content-Type: application/json
   Prefer: return=minimal
   ```
   Body: array of 50 location objects.

   Map enriched JSON fields to DB column names:
   - `lat` → `latitude`
   - `lng` → `longitude`
   - All other field names already match the schema.

   Always set `published: false` on every record regardless of enriched JSON value.

5. **After each successful batch**, append the imported slugs to `scripts/data/imported.log` (one slug per line).

6. **Print progress** after each batch: `Imported batch {N}: {N} records`

7. **Final summary:**
   ```
   ✓ Import complete
   - {N} locations imported
   - {N} skipped (already imported)
   - {N} failed validation (check output above)
   
   Next step: Go to Supabase dashboard, review the imported records,
   then run UPDATE to flip published=true for approved locations.
   ```

## Notes

- Never set `published: true` automatically — always requires manual review
- If the Supabase insert returns a conflict error (duplicate slug+state), skip and log it
- The `id` field will be auto-generated by Supabase — do not include it in the insert payload
- After import, you can flip published in bulk via Supabase SQL:
  ```sql
  UPDATE locations SET published = true
  WHERE state_slug = 'arizona' AND published = false
  AND created_at > NOW() - INTERVAL '1 hour';
  ```
